1 Initial Ideas

First passes after watching the Harbor video — figuring out where my work needed to focus.

  • Watched the harbor video, didn't go in-depth, but it seems like it can handle running a task if the dataset is present, and it will invoke my grader as well.
  • I need to focus on:
    • Having an automated way to create the dataset
    • Have a grader tool
    • Integrate all of this with claude and modal and harbor so that it all works nicely together
  • Seems like a nice idea to have the "dataset creator" and the "output grader" have some initial source of truth — will be easy to grade that way.
    • For example, assume I have the info of how a website looks in some format F, so I have a list like F1, F2, F3….
      • The creator will convert these into a collection of screenshots data → D1, D2, D3 and so on
      • The agent gets Dx and gives Ox (output code)
      • Grader gets Fx (and maybe also Dx) and Ox and evaluates Ox based on whether it fulfills the requirements
    • The problem boils down to deciding the format F.
      • Could it just be a prompt?
        • The "creator" creates a website from the prompt and then screenshots it → using playwright python tool
        • The "grader" can simply check if the website is according to the prompt or not
        • The weakness here might be over-reliance on AI →
      • Once I decide what is F → how do I generate many different Fx?
      • F can be a prompt along with a randomly generated JSON description of various features in the website → the grader must use Dx to evaluate Ox.

Generate a website based on a list of given requirements:

{
        "Type" : "Social Media"
        "Topic" : "finance",
        "button_density" : "0.8",
        "corner_pointyness" : "0.3",
}

Can randomly create multiple JSON descriptions of files using an RNG → these JSON descriptions contain some fixed fields and some temperature-like knobs and dials which forces AI to diverse samples instead of making all similar websites.

2 Generate Task

Turned the idea into a real Harbor task — generating the dataset of websites end-to-end.

Created a harbor task with the above idea, a gen.py is called on environment setup which creates the json — the resulting json is fed to claude_code with the instructions.md to create the website. The verifier is a dummy which always gives score=1.0, but it converts the html files to screenshots by using playwright chromium. Multiple trials of this same task will generate different looking websites. (run with -k 20 for 20 websites.)

  • Ran these tasks on Modal to finish quickly and not overheat my laptop.
  • Here I first made websites using low-effort Haiku to make sure everything is working and the generated websites are diverse.
  • To debug and inspect → I ended up making some automations and dashboards:
    • Essentially a script keeps watching task success and collects the artifacts into a folder as the trials keep on getting completed.
    • I spin up a webserver which has a dashboard which shows a slideshow of the screenshots generated by the finished tasks along with thumbnails.
Once I was sure that the output looked fine, I ran it with Opus on xhigh effort (max effort sometimes overthinks and pollutes its own context).

3 Clone Task Template

One template task, many stamped instances — one per generated website.

Need to create multiple clone tasks → once for each of the generated websites. So I just created a template-task which is like a boilerplate — then a script run_tasks basically manages everything; it uses the template to create multiple tasks and runs them all on Modal.

  • This task_template is essentially just "you are given some screenshots, make a clone website" along with some extra instructions to make sure it uses only HTML and CSS.
  • The verifier here uses the Websight metric (mean of OCR_f1, lpips and ms_ssim) → I decided this by evaluating some outputs over different metrics (implemented in evaluators in the repo, and also evaluator_task to run in parallel on Modal) → and I realized that "websight" is the metric that is closest to my human perception. Used it to compare 2 screenshots → the final score of the entire website is just the mean of the scores from all the individual pages.
  • Here also, I first generated everything with low-effort Haiku to make sure everything is alright in the pipeline.
    • Again I made another dashboard to keep "watching" the results as the trials kept finishing.
Once I was sure that the pipeline is fine, ran it with Opus max effort.

7 Power Mean Across Pages

Another thing that I saw while looking at the results manually → taking the mean over multiple pages seems to "forgive" one bad page. In fact, many times, sparse or un-complicated pages tend to help the website "win".

  • To tackle this problem → instead of taking the average of the scores across pages, I decided to take the Generalized Power Mean with power less than 1 (I tried out different values to get rid of "bad apples") → intuitively, a value less than 0.25 seemed like what I needed for this task → especially since my scores are in the range of [0, 1].
Ended up making a metric mae_pm_0.05 → MAE per page, then taking a generalized power mean with power 0.05 across pages.

8 Animated Pipeline

After the static HTML+CSS clone pipeline was working end-to-end, I started on the animation bonus task.

Generate_animated task

  • Reused the same gen.py + knobs-and-dials approach but in the instructions, I asked it to use animations.

Capture format

  • Each page gets rendered via a custom Playwright recipe → 6 keyframes (full page) at exactly t = 0.0, 0.5, 1.0, 1.5, 2.0, 2.5s from first paint, plus a continuous WebM recording of the full 3s window.
  • All of the above is done with Playwright Chromium.

Clone_animated_template

  • Same stamping pattern as the static clone task → run_tasks has an ANIMATED=1 flag that swaps the template to use clone_animated_template task.
  • The agent's instructions are in the same spirit as the static task → one /app/pages/<slug>.html per slug.
  • Inspect_results (view_screenshots and view_outputs) picks up the animated layout automatically → per-slug comparison views instead of flat png pairs.

Evaluator for animation

  • Didn't research much around this — just used MAE for every keyframe and combined the results with pm_0.05.
  • Also → I wanted to make sure that the motion happens in the similar pixels in target and candidate webpage → so I made a temporal metric and averaged the static and temporal metrics → the temporal metric is basically just "MAE over the delta of keyframes" from target and candidate.
The nice thing here is that I managed to make the pipeline run with the same 2 commands — just need to pass ANIMATED=1 in the env vars.
